Phase 4a — L2 + L3 validation on pair 11119

End-to-end pipeline run on one (pair, skycell) before scaling

What Phase 4a does

Before scaling to all 5 pairs, validate the full L2 → L3 → (Phase 5) source detection chain on one (pair, skycell). Concretely, for pair 11119 (skycell 000m18x58y45):

  1. Build a combined truth catalog (stars + galaxies merged, mixed type column — romanisim ingests both in one file).
  2. Emit 14 romanisim-make-image commands (output/detection/sims_11119.script), one per (pointing, SCA) pair from Phase 2, each with a deterministic CRC32-of-basename seed so every dither draws an independent noise realisation.
  3. Run the sims 4-way parallel. Output L2s in output/detection/l2/ (2.8 GB for 14 files).
  4. Run skycell_asn --product-type full on the 14 L2s to get one association JSON per overlapping skycell.
  5. Run MosaicPipeline on the selected-skycell association.
  6. Inspect depth, weight, image, truth-to-image correspondence.

Wall time: 14 sims × ~5.8 min / 4-way = ~23 min for stage 02, plus ~3 min each for stages 03 + 04 = ~30 min end-to-end.

Depth anomaly — Phase 2 galsim-vs-CRDS WCS mismatch

Phase 2 predicted skycell 000m18x58y45 would reach depth 6 (all 6 pointings contributing via at least one SCA). The real skycell_asn output contradicts that: only 4 L2 files overlap x58y45, from only 4 of the 6 pointings (exposures 2 and 3 of each pass; exposure 1’s SCAs 2 and 3 land ~0.1° west of the skycell, outside its footprint).

Cause: Phase 2 computed SCA corners via galsim.roman.getWCS, which uses GalSim’s analytical distortion model (SIAF-based). The L2 sims used --usecrds, which applies the CRDS distortion reference — a per-SCA tweak that can shift corners by several arc-minutes. Near skycell edges (4.6’ cells), that’s enough to flip the overlap decision.

What this means

  • For Phase 4a: depth-4 instead of depth-6 in the mosaic. Still enough to exercise the pipeline end-to-end, but the expected detection threshold is ~0.2 mag shallower.
  • For Phase 4b (scale to 5 pairs): we need to fix Phase 2 before re-selecting skycells. Likely fix: replace galsim.roman.getWCS with romanisim.wcs.get_wcs(..., usecrds=True) for the overlap analysis, which exercises the same CRDS path the L2 sim uses.

What works, what doesn’t

Works:

  • L2 sim pipeline: 14/14 sims succeed, no CRDS hiccups, seeds produce independent noise realisations (log files clean).
  • skycell_asn: emits valid associations for 22 skycells. Target skycell’s asn has 4 members, consistent with actual L2 coverage.
  • MosaicPipeline: runs on the asn, produces a 410 MB coadd (image + weight + context arrays), zodi background and sigma match expected.
  • Truth-to-image mapping: the brightest star (mag 23.0) shows as a clear PSF in the mosaic at exactly the injected RA/Dec. Fainter sources and extended galaxies show at their expected positions.

Doesn’t (to fix before Phase 4b):

  • Phase 2 overlap analysis uses galsim’s internal distortion model, not the CRDS distortion applied by romanisim-make-image --usecrds. The two disagree by several arcmin per SCA, flipping near-skycell-edge decisions. Effect: Phase 2 predicted depth 6 but reality is depth 4 for the representative skycell.

Checkpoint

Phase 4a success criteria met: pipeline works end-to-end on one (pair, skycell). Blocker for Phase 4b / 5 full scale-up: the Phase 2 bug needs fixing before committing to 5 pairs × 5 skycells × more sims. Proposed fix: replace galsim.roman.getWCS with romanisim.wcs.get_wcs(image_dict, usecrds=True) in 02_skycell_overlap.py, re-run Phase 2, re-select skycells, then regenerate Phase-3 catalogs at the new skycell positions before Phase 4b.
