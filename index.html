roman_l2_job

Roman WFI L2 → L3 simulation pipeline + photometry recovery test

Author

Nikhil Padmanabhan

Published

April 19, 2026

What this is

roman_l2_job is a self-contained pipeline that simulates Roman WFI Level-2 calibrated images over a chosen footprint, builds Level-3 skycell coadds and source catalogs from them, and measures how faithfully the recovered catalog reproduces the input catalog — a photometry recovery test. It was built to support simulated grism observations centered on a Wide-Field2 sub-region near (RA=10°, Dec=0°), but generalizes to any F158 footprint and any input source catalog.

The code, configs, and this report all live in the roman_l2_job GitHub repo.

Contents

  • Pipeline — architecture, stages, data models, design choices (why certain scripts are vendored, how CRDS is hydrated, etc.).
  • Results — numbers from the full 324-sim run over the 6-visit / 2-segment footprint: depth distribution, completeness vs magnitude, flux residual, astrometric residual, PSF vs aperture photometry.
  • Discussion — what the numbers mean for the intended science use (grism support), with notes on depth geometry, Kron flux bias, and zodiacal background.

Headline numbers

Note

From the full run (6 visits × 3 dithers × 18 SCAs = 324 F158 sims, 320 skycell coadds + catalogs, 4 h 5 min stage-04 on 4 m5a.2xlarge cores):

  • Median 50%-completeness (AB): 23.95 over all skycells, 24.05 in the deep-pixel subset, peaks at 24.50 in the deepest skycells.
  • Kron flux residual: +0.17 mag (recovered − input), unchanged across depth — it’s a Kron-photometry property on extended sources, not a pipeline calibration issue.
  • Astrometric residual: ~0.017″, sub-coadd-pixel across every skycell.
  • Depth distribution: only even integers appear in the per-pixel depth (2, 4, 6, 8), because pass 15 and pass 16 re-observe each pointing at the identical dither pattern — sum is always 2× an integer.
  • Field geometry: only ~25% of skycells within 0.5° of the field center hit the intended design depth; the full run is more a union of three visit-cluster cores than a uniform 1° disk.
