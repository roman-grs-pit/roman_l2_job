Full run — discussion

A handful of findings from this run are worth flagging for whoever uses the pipeline next — whether to follow up, to adjust the design of the next campaign, or to keep in mind when consuming the catalogs.

Footprint non-uniformity around (10°, 0°)

Our target is a grism reference point near (RA = 10°, Dec = 0°), which sits at the northern survey boundary of the HLWAS simulation used to build the pointings. The default APT-planned tiling (at least in the version of the HLWAS .ecsv we filtered) does not uniformly cover this region — the filter_pointings.py cone pulls out three separated visit-clusters (segments 490 and 521) with shallow between-cluster gaps:

  • Only ~25% of skycells within 0.5° of the target reach the design depth of 6.
  • Only ~52 of 320 skycells have median_depth == 6; the bulk sit at median depth 2.

Options if we want better coverage of this field:

  • Move the default. If the grism target doesn’t have to be exactly (10°, 0°), picking a point ~0.5°–1° south would put it inside the survey interior where tiling is uniform. Downside: we’d lose any continuity with prior analyses tied to the current coordinates.
  • Add pointings. Two visits with the default dither pattern around (10°, 0°) would roughly double the existing coverage (+108 sims on top of the current 324) and fill the immediate neighbourhood without moving the target.
  • Go to a 3×3 grid. Nine pointings centered on the target is a clean geometry; budget is 9 × 3 × 18 = 486 new sims (~16 hr of stage 02 at current pace). To limit L3 work after that, we could run stages 04 and 05 only on skycells with sufficient asn-member count — e.g. len(asn.members) >= 14 correlates well with per-pixel depth ≥ 6 in the current data. That filter alone would drop stage-04 load by ~80% on the 3×3 footprint.

Whichever route we pick, the pipeline side of that work is straightforward: edit the config (new pointings.region centre, wider radius_deg for a grid) and re-run from stage 01. The skip-if-exists guards mean cal files from the current run will be reused for any overlapping pointings.

Zodiacal background: in MJy/sr, close to the Roman reference

What we measured. Sigma-clipped median on an F158 cal file (output/cal/r0000101015521001006_0001_wfi01_f158_cal.asdf), with detected sources masked (5σ threshold, ≥5 connected pixels):

  • 0.65 DN/s/pix on the raw data array. romanisim.image.make_l2 documents this array as ramp slopes in DN/s.
  • 0.50 MJy/sr via the cal file’s meta.photometry.conversion_megajanskys = 0.77.
  • 1.28 e⁻/pix/s if we multiply DN/s by the WFI01 CRDS gain reference (median 1.99 e⁻/DN).

The 12-file sample averaged across visits and SCAs (median of medians) gives 0.67 DN/s/pix, i.e. 0.52 MJy/sr — consistent field-to-field within a few percent.

What we compared against. A Pandeia ETC run at (RA=10°, Dec=0°), F158, single 107.52 s exposure, returns an input zodi of 0.38 MJy/sr and a total background of 10.71 e⁻/s in a 51.93-pixel aperture → 0.206 e⁻/pix/s.

Result in MJy/sr. 0.50 vs 0.38 → 1.32× higher. That’s a modest model-vs-model difference between romanisim’s zodi model and Pandeia’s at the same coords and date, worth ~2.5·log₁₀(√1.32) ≈ 0.15 mag of 5σ point-source depth.

Result in e⁻/s. 1.28 vs 0.206 → 6.3×. This is not a real background excess. Cross-check: we aperture-photometered 30 isolated input point sources at AB 18.5–20.5 on the same cal file (3-pix circular aperture, annulus-subtracted sky, EE-corrected), multiplied by gain to get e⁻/s, and compared to the ETC zeropoint implied by its mag-24 example (1.42 e⁻/s in a 0.2″ aperture). The median measured/predicted ratio was 6.7× — matching the sky ratio to within 5%. Both source and sky carry the same offset in the e⁻/s reporting, so the source/sky ratio — and therefore SNR at fixed AB magnitude — is internally self-consistent within romanisim.

The likely origin of the 6–7× e⁻/s discrepancy is a convention difference between romanisim and Pandeia somewhere in the chain {bandpass integral, effective area, throughput, gain} — the two codes have to agree in MJy/sr (they roughly do) and in AB magnitudes (that we have not independently verified), but the e⁻/s-per-MJy/sr bridge between them is not constrained to match.

Cross-check against romanisim’s own sky model. romanisim.image (line ~697) injects sky using galsim.roman.getSkyLevel, scaled by (1 + stray_light_fraction) and with thermal_backgrounds[H158] added per pixel. Calling that chain directly at (RA=10°, Dec=0°), 2026-01-01:

term value
getSkyLevel (zodi only) 95.01 e⁻/s/arcsec²
× (1 + stray_light_fraction = 1.1) 104.51 e⁻/s/arcsec²
× pixel area (0.11² = 0.0121 arcsec²) 1.265 e⁻/s/pix
+ thermal_backgrounds['H158'] = 0.040 e⁻/s/pix 1.305 e⁻/s/pix

That matches the cal-file measurement of 1.298 e⁻/s/pix to 0.5%, confirming the sim is faithfully injecting exactly what romanisim’s sky model predicts. Translating back to MJy/sr via the same romanisim-internal get_abflux(F158) / 3631 Jy × pix_area_sr conversion used above, 1.305 e⁻/s/pix corresponds to 0.493 MJy/sr, versus 0.502 measured — again consistent. So the ~30% MJy/sr gap against Pandeia’s 0.38 is a genuine zodi-model disagreement between the Hirata tables galsim uses and Pandeia’s model, not a sim setup issue.

Gain consistency, as a side effect. getSkyLevel is gain-free (it returns e⁻/s/arcsec² from a direct table lookup), but our cal-side number is DN/s × gain with gain = 1.994 e⁻/DN from the WFI01 CRDS reference (roman_wfi_gain_0022.asdf). The 0.5% agreement therefore verifies that the gain we multiply by on read matches the gain romanisim divided by when writing the DN/s slopes — a consistency check, not an absolute gain validation. (If the CRDS gain were physically off by 2× both sides would be wrong by 2× and still agree; that would need a separate absolute test, e.g. a ramp Poisson-noise consistency check.) In practice this is enough to trust DN/s × gain_CRDS as the correct way to recover e⁻/s from the cal file for any downstream analysis.

Upshot. Zodi is not responsible for the observed 0.5–1 mag depth shortfall. The earlier wording in this section — “3.4× the polar minimum → 0.67 mag of depth loss” — compared romanisim’s e⁻/s against a Pandeia-convention e⁻/s reference, which we now know is an apples-to-oranges comparison, and is retracted. The depth shortfall is still real and the explanation is still open.

psf_flux appears to be in different units than aper08_flux / kron_flux

For input PSF sources, we expect psf_flux ≈ aper08_flux (both capture total flux for point sources). Measured offset: psf_mag − aper08_mag = +0.656 mag (input-PSF sources, psf_flags = 0). The matching kron − aper08 is +0.07 for the same sources, so kron and aper08 agree well.

Grepping romancal.source_catalog.psf, psf_flux comes straight from a photutils.psf.PSFPhotometry fit on image_model.data with the flux_fit column renamed — no unit conversion is applied on the way out. aper*_flux and kron_flux in the same catalog output, in contrast, are in nJy (we confirmed kron_flux round-trips with kron_abmag via the standard AB formula to 1 part in 10⁴).

The systematic offset matches the median CRDS-gain across the 18 WFI SCAs (1.869 e⁻/DN → 2.5 × log₁₀ = +0.679 mag) to ~2%. Our working hypothesis is that psf_flux is in DN/s-like units while aper_flux and kron_flux have been converted to nJy. The 2% residual may come from (a) per-SCA gain variation, since L3 coadds are weighted combinations of contributing L2s, or (b) PSF-wing truncation in the 8-pixel aperture, which would push the sign the other way, so probably (a) dominates.

Consequences for downstream science:

  • Do not use psf_flux as a standalone total flux estimator until the unit chain is verified. Use kron_flux, which is internally self-consistent in this catalog, for flux comparisons.
  • Worth filing upstream against romancal to either (a) apply the gain correction to psf_flux in the same pipeline stage that converts aperture/Kron fluxes, or (b) document the difference in unit explicitly.

Depth quantization to even integers

For context on the Results depth-distribution plot — per-pixel depths in this run occur only at even integers (2, 4, 6, 8). That is not a coverage artifact: passes 15 and 16 of the HLWAS design re-observe each visit with identical (RA, Dec, PA) and dither offsets, so a pixel sees depth k in pass 15 and depth k in pass 16 — the sum is always 2k. The smoke config (1 visit × 3 dithers, no second pass) did see odd depths 1/2/3; the full config shouldn’t.

Kron flux bias (+0.17 mag)

The median recovered Kron − input mag of +0.17 mag is stable across depth, skycell, and the smoke-vs-full comparison. It’s a well-known Kron property on extended sources — Kron radius is self-sizing and catches ~90% of a Sérsic profile’s light, not 100%. For input PSF sources the Kron residual is much smaller (median +0.07 mag on a deep skycell check). This is a photometry property, not a pipeline calibration issue.

If you want an unbiased recovered flux for galaxies specifically, segment_flux or a larger aperture (e.g. aper16_flux) would be the better choice.
